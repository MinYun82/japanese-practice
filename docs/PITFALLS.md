# 地雷經驗總結 — 日文學習工具開發

> 建立時間：2026-04-04
> 適用範圍：為年長初學者製作互動式學習工具

## 地雷 1：Web Speech API 的瀏覽器差異

**問題**：`speechSynthesis.getVoices()` 在 Chrome 首次呼叫時回傳空陣列（非同步載入），Safari/iOS 有不同限制。

**解法**：
1. 監聽 `voiceschanged` 事件
2. 設 3 秒超時 fallback
3. iOS Safari 需在 `cancel()` 後加 50ms 延遲再 `speak()`
4. 首次必須由使用者互動觸發（autoplay 限制）

**程式碼參考**：`src/speech.js`

---

## 地雷 2：認知負荷設計 — 內容多不等於學得好

**問題**：把所有內容塞在一個頁面，67 歲使用者反而更混亂、沒成就感。

**解法**：
- 每日限量（dailyLimit）控制在 8-15 個項目
- 用「概念 → 對比 → 翻卡 → 測驗」漸進路線
- 結果頁顯示「具體進步」而非只有分數
- 字體至少 48px（日文字）、20px（說明文字）、按鈕 48x48px 最小

**教訓**：為老人設計要「做減法」，不是「做加法」。

---

## 地雷 3：學習素材與課程進度脫節

**問題**：每次根據媽媽口頭回饋才做練習頁，反應總是慢一拍，且無法累積。

**解法**：
- 資料驅動架構：一個 JSON 設定檔 = 一週練習
- 編輯 JSON 10 分鐘就能產出完整練習
- 設定檔可放 compareWith 指定對比組
- 設定檔跟教材進度對齊（大家的日本語）

**教訓**：先建框架，再填內容。不要每次重做 HTML。

---

## 地雷 4：間隔重複的過度工程

**問題**：SRS 演算法可以做得非常複雜（SM-2、Anki 等級），但初學者不需要。

**解法**：
- 極簡間隔：mastery 1=1天, 2=2天, 3=4天, 4=7天, 5=14天
- 掌握度計算只看「正確率 + 練習次數」
- 不需要 ease factor、interval modifier 這些進階參數

**教訓**：KISS 原則 — 對 67 歲初學者而言，「有複習」比「完美排程」重要 100 倍。

---

## 地雷 5：LocalStorage 資料損壞

**問題**：localStorage 可能因為使用者清除瀏覽器資料、容量滿、或 JSON parse 錯誤而損壞。

**解法**：
- 所有 `JSON.parse()` 都用 try-catch 包
- 損壞時回傳預設空物件，不要 crash
- 定期進度會自然重建（因為間隔重複會重新記錄）

**教訓**：進度丟失很傷感情，但比 app crash 好。用防禦性程式碼。

---

## 地雷 6：干擾選項的品質

**問題**：測驗的 4 選 1 如果干擾選項太離譜（例如用片假名當平假名題的選項），使用者不用真的會就能猜對。

**解法**：
- 優先從同類型（同為濁音、同為 hiragana）挑干擾項
- 不夠時才跨類型補
- 排除與正確答案完全一樣的字

**教訓**：干擾選項要「像但不是」，才有練習效果。

---

## 地雷 7：教材對齊的重要性

**問題**：自製練習如果和老師課堂教的順序不一致，會讓學習者混亂。

**解法**：
- 確認使用的教材（大家的日本語 初級 I，大新出版社）
- 週設定檔跟課堂進度同步
- 老師教什麼就練什麼，不要「超前部署」

**教訓**：輔助工具的角色是「鞏固」不是「教新的」。跟著老師走。

---

## 地雷 8：週計畫拆成每日 — 不能平均分配

**問題**：把一週 15 個學習項目平均分成 7 天（每天 2 個）看似合理，但實際上 Day 1 需要先建立概念，Day 7 需要總複習，中間才是主要學習時段。

**解法**：
- Day 1-2：認識 + 聽（翻卡、概念說明），不考試
- Day 3：第一次小測驗（鞏固前兩天）
- Day 4-5：深化 + 單字 + 對比
- Day 6：綜合翻卡複習
- Day 7：總測驗
- 每天都混入前幾天的複習項目（滾動式）

**教訓**：學習節奏不是「平均分配內容」，而是「漸進加深認知」。

---

## 地雷 9：不可預測的課程進度

**問題**：老師的教學進度無法預知，不能預先建好所有週設定。

**解法**：
- 採用「事後反應式」工作流：上完課 → 告訴 Claude 學了什麼 → Claude 產出 JSON
- JSON 格式標準化（含 dailyPlan 7 天拆分），可重複使用模板
- 練習框架與內容分離，換內容不改程式碼

**教訓**：工具要適應使用者的節奏，不是要求使用者適應工具。

---

## 地雷 10：dailyPlan 中的 mode 字串設計

**問題**：一開始想用陣列表示多模式 `["concepts", "flipcard"]`，但 JSON 解析和 UI 邏輯都變複雜。

**解法**：
- 用 `+` 分隔：`"concepts+flipcard"`
- 前端只看第一個模式啟動，其他模式在結束時提供「接下來」按鈕
- 簡單字串比物件容易維護

**教訓**：KISS — 能用字串解決的不要用物件。

---

## 地雷 11：日期硬編碼導致每日規劃失效

**問題**：`weekConfig.date` 設成「未來日期」或「今天」，導致 `getDayNumber()` 永遠回傳 Day 1，使用者連續多天看到相同的內容（只有が行 5 個字）。

**解法**：
- `date` 必須設為**實際上課的那一天**（週六），不是「預計開始練習的日期」
- `getDayNumber()` 用本地日期 (`new Date(year, month, day)`) 計算，避免 `new Date("2026-04-04")` 被當成 UTC 造成時區偏移
- 超過 7 天 → clamp 到 Day 7（總複習），不能回傳 0 或負數

**教訓**：日期是最常被忽視的 bug 來源。手動設定日期時，要驗證「今天打開看到的是第幾天」是否合理。

---

## 地雷 12：資料結構不統一 — 程式碼找不到項目

**問題**：`week2-data.js` 用 `dakuon`、`handakuon`、`sokuon` 三個獨立陣列存放學習項目，但 `resolveItems()` 只從 `weekConfig.newItems` 查找。結果 dailyPlan 裡的 item ID 在 newItems 裡找不到，練習內容就是空的。

**解法**：
- **統一資料結構**：所有週設定檔的學習項目都放在 `newItems` 陣列中
- 用 `type` 欄位區分類別（`"type": "dakuon"`、`"type": "handakuon"`）
- 不要發明新的 top-level key，讓消費端（resolveItems/QuizGenerator）不需要特殊處理

**教訓**：資料結構是 DDD 的地基。如果 Entity 的儲存格式不統一，所有 Service 層都會出錯。新增內容類型時，擴展 `type` 欄位，不要新增陣列。

## 地雷 13：新舊學習內容混合洗牌

**問題**：每日練習的翻卡把「今天的新項目」和「複習項目」混在一起隨機洗牌。使用者點「促音單字」進去，第一張卻看到濁音的「ざ」（複習項目），以為內容路由錯了。

**根本原因**：`generateFlipCards` 用 Fisher-Yates 對所有項目統一洗牌，不區分新舊。

**解法**：
1. 新項目和複習項目**分開洗牌**，新的在前、複習在後
2. 複習卡片加上「複習」標記，視覺上區分
3. `DailyPlanner.resolveItems` 回傳 `{ newItems, reviewItems }` 分開的結構

**通用教訓**：任何練習系統的項目排序，都應該把「當前學習目標」放在最前面，輔助/複習內容放後面。使用者的注意力和期待在開頭最高。

---

## 地雷 14：翻卡缺少操作提示

**問題**：67 歲使用者不知道要點卡片翻面看答案。卡片正面只有假名和喇叭按鈕，沒有任何提示文字。

**解法**：在卡片正面底部加「點卡片翻面看答案」文字。

**通用教訓**：對非數位原住民的使用者，每個互動元素都需要明確的操作說明，不能假設使用者會自己探索。

---

## 地雷 15：跨週複習項目缺少輔助資訊

**問題**：Week 2 複習 Week 1 的假名時，`findById` 只從當週 `newItems` 和 KANA_POOL 查找。KANA_POOL 的資料沒有 `chineseHint`/`zhuyinHint`，導致複習卡片翻面後缺少中文諧音和注音提示。

**解法**：`findById` 擴展查找範圍：當週 → 所有週 ALL_WEEKS → KANA_POOL fallback。

**通用教訓**：跨模組引用資料時，要確保取得的是**完整**的物件，不是精簡版。特別是 UI 需要的輔助欄位，在 fallback 路徑中很容易遺失。

## 適用於其他專案的通用經驗

以上地雷中，以下幾點可廣泛適用：
- **地雷 1**（Web Speech API）：任何需要語音的 web app
- **地雷 2**（認知負荷）：任何面向非技術使用者的工具
- **地雷 4**（過度工程）：任何 MVP 階段的產品
- **地雷 5**（localStorage）：任何依賴瀏覽器儲存的 app
- **地雷 6**（干擾選項品質）：任何學習類測驗工具
- **地雷 8**（學習節奏設計）：任何教育類產品的排程設計
- **地雷 9**（不可預測進度）：任何需要跟外部節奏同步的工具
- **地雷 11**（日期硬編碼）：任何依賴日期計算的排程系統
- **地雷 12**（資料結構不統一）：任何資料驅動的應用
